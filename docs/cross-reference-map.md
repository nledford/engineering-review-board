# Cross-Reference Map

This is a routing overlay for relationships between existing skills. It is not a
second inventory; keep full skill purposes, category boundaries, and coverage in
[`skill-taxonomy.md`](skill-taxonomy.md).

## Row Shape

Use one row per decision point where routing can involve more than one skill or a
validator-enforced relationship.

| Column | Meaning |
| --- | --- |
| Route | Short task surface or conflict being routed. |
| Primary skill | The first skill that owns the work. Use one primary unless the route is explicitly review-composed. |
| Related required skill(s) | Skills that must be loaded, linked, or handed off to for this route. Use `load:`, `link:`, or `handoff:` prefixes. |
| Optional companion skills | Skills to add only when local evidence matches their trigger. |
| Should-trigger examples | Short phrases that should activate the row. |
| Should-not-trigger examples | Near misses that should route elsewhere or avoid the row. |
| Validation source notes | Where enforcement or authority comes from: validator constant, taxonomy scenario, repository docs, or first-party skill frontmatter/body. |

Notation:

- `load:` agent routing requirement for the active task.
- `link:` repository validation expects a `SKILL.md` cross-link.
- `handoff:` use the named skill before reporting that class of finding.
- `—` means no required companion for the row; optional companions may still
  apply.

Skill names in this matrix route reusable procedure; they are not runtime Task
agent IDs and do not grant delegation authority. When a route crosses into an
OpenCode agent or command, use the exact registered ID and boundaries documented
in the [engineering agent governance guide](engineering-agent-governance.md).

## OpenCode Runtime Handoff Overlay

This compact overlay is not a skill inventory. Exact agent and command definitions
and the [engineering agent governance guide](engineering-agent-governance.md)
remain authoritative.

| Route | Handoff |
| --- | --- |
| Primary-agent handoff | Authority follows the primary agent selected for the current user turn. Earlier turns remain context but do not transfer permissions. Top-level primary roles may hand off in the same conversation: `/address-review` re-anchors the current command turn to the Engineering Lead, treats earlier ERB output as read-only advisory context from a different primary agent, and re-evaluates it before ordinary implementation. `/brainstorm` re-anchors the current command turn to the read-only Engineering Review Board for solution exploration without inheriting another primary agent's identity or permissions. `/optimize-prompt` re-anchors the current command turn to the Engineering Lead for read-only prompt optimization. `/semver` re-anchors the current command turn to the Engineering Lead for explicit audit, apply, or local-tag workflows. `/root-cause-analysis` re-anchors the current command turn to the read-only Engineering Review Board for causal analysis and an agent-challenged repair proposal. `/consult-plan`, `/create-plan`, `/update-plan`, and `/start-plan` re-anchor the current command turn to the Plan Orchestrator without inheriting Board or Lead identity, permissions, or authority. The ERB remains read-only when selected. Use a fresh conversation when formal contextual independence matters. |
| External directory audit | The current human request or a bounded Task packet names one exact external root. Approval is ask-gated for each invoked agent or subagent and never transfers through Task; the Plan Orchestrator remains denied. The external root is supplied scope rather than the active workspace, read-only roles keep edit denial, path-bearing shell or Git inspection needs separate Bash permission, literal roots remain machine-local, and reports sanitize private paths and contents. |
| Direct delivery | The Engineering Lead may implement directly whenever scope, safety, and validation are adequate. Complexity may justify recommending planning, never automatic plan creation or `/start-plan`. |
| Read-only solution exploration | A human may choose top-level `/brainstorm` when multiple credible engineering paths need comparison before implementation. The selected Engineering Review Board loads `brainstorming`, uses direct Board analysis by default, and delegates only when a minimum sufficient specialist panel could materially change the recommendation or resolve an evidence gap. It returns options, tradeoffs, an advisory recommendation, validation needs, and the next human-controlled route without editing, delegating implementation, creating plans or state, staging, committing, or beginning the recommendation. Active unexplained failures route to investigation first; a single credible path should be reported rather than padded with invented alternatives. Direct implementation requires a separate explicit human request selecting the Engineering Lead, optionally through `/address-review`. |
| Read-only prompt optimization | A human may choose top-level `/optimize-prompt` to rewrite a coding-agent prompt without executing it or editing a referenced source file. The selected Engineering Lead loads `prompt-engineering-review` and `review-verification-protocol`, delegates exactly one bounded read-only analysis and rewrite to `prompt-critic`, then verifies and owns the final response. The command preserves material constraints and exact identifiers, verifies repository-local names when practical, and returns a copy-ready replacement with material corrections, assumptions, and verification. Reusable `SKILL.md` contracts route to `create-agent-skill`. |
| Semantic version workflow | A human may choose top-level `/semver` with exactly one mode. `audit` loads `semantic-versioning` and remains read-only; `apply` re-audits fresh evidence, changes only version metadata and required mirrors, and does not commit or tag; `tag` also loads `git-workflows` and may create one local release tag only when a clean committed `HEAD` already contains the audited target version. No mode pushes, publishes, deploys, or decides final release readiness. |
| Read-only root-cause repair proposal | A human may choose top-level `/root-cause-analysis` after an event or direct cause has enough evidence for causal analysis. The selected Engineering Review Board loads `root-cause-analysis`, `brainstorming`, and `review-verification-protocol`; confirms the symptom-to-root-cause and control-gap chain; delegates distinct read-only questions to every decision-relevant specialist; synthesizes the smallest safe root-cause repair; and then requires `adversarial-reviewer` to challenge the final proposal. An unconfirmed root cause, missing evidence, or unresolved material objection stops clearance. **Recommended for human consideration** is advisory evidence only: the command makes no changes, and direct implementation requires a separate explicit human request selecting the Engineering Lead. |
| Code documentation | For audit-only code comments, docstrings, embedded API docs, examples, or documentation tests, the Lead or ERB may delegate exact source scope to `documentation-critic`. A code-only assignment treats standalone Markdown as evidence rather than a review or edit target. Requested corrections remain implementation owned by the Lead or one bounded `implementation-worker` unit, with `documentation-engineering` and matching language/testing skills providing procedure rather than authority. |
| Read-only consultation | The Engineering Lead or ERB may recommend top-level `/consult-plan` with the reason, trade-off, and proposed scope. The separate primary Plan Orchestrator provides non-mutating advice without inspecting `.erb/plan-state.json`, creating a plan, delegating, authorizing, or executing work. The human controls the route. |
| Explicit plan creation | A human explicitly chooses top-level `/create-plan`; the Plan Orchestrator creates one `.erb/plans/<slug>.md` plan or, only for genuinely separately managed work, a `.erb/plans/<subject>/<NN>-<slug>.md` series, then writes the selected plan path to `.erb/plan-state.json`. Creation is plan-only. |
| Active plan update | A human explicitly chooses top-level `/update-plan <exact-plan-path>`; the Plan Orchestrator updates only that active canonical plan with the smallest exact-content patch, conservatively retains or resets checked evidence, leaves `.erb/plan-state.json` unchanged, and stops without delegation, implementation, or execution. Completed plans remain immutable, and later execution requires a separate `/start-plan`. |
| Plan artifact commit | After the Plan Orchestrator creates and validates a plan, an explicit current human commit request may authorize the selected Engineering Lead to stage and commit only the canonical plan Markdown. This exception grants no plan edit, state, or execution authority; `.erb/plan-state.json` remains excluded. Every staged path is freshly derived, literal, contained, and runtime approval-gated. |
| Conversational plan replacement | A current explicit human instruction to split or replace one unambiguous canonical plan is plan-only authority to create at least two successors and retire that source without another deletion confirmation. The Plan Orchestrator creates and re-reads every successor, re-reads the source, then retires only the exact source with an exact-content edit patch. Review advice alone cannot authorize this; no registry or retained contract history is involved. |
| Plan selection and resume | A separate human chooses execution-only `/start-plan <existing-plan-path>` or resumes the selection in `.erb/plan-state.json`. An explicit valid path repairs missing, invalid, or stale state. Active means any TODO or Verification checkbox remains unchecked; the first unchecked checkbox is current. A fully checked plan reports that it has already been implemented. |
| Planned implementation delegation | Each new Worker Task receives a self-contained packet with the canonical plan and TODO context, relevant plan constraints, owned scope, numbered acceptance criteria, validation, and stop conditions. One-at-a-time limits active concurrency, not correction attempts: the Plan Orchestrator treats a Worker return as evidence, maps every criterion to fresh source, diff, and validation evidence, and resumes the same Task child with enumerated evidence gaps, blocked criteria, observed and required results, exact correction scope, validation to rerun, and unchanged constraints before checking the TODO. Status-only references to findings are incomplete; the Worker blocks rather than guesses. |
| Concurrent plans | Selection state is not concurrency control. The latest explicit valid plan path wins, and the workflow does not block, coordinate, or recover another plan execution. |
| Plan-only operations | Active plan bodies are immutable by default and during execution except for evidenced checkbox advancement. A current `/update-plan <exact-plan-path>` request may update one active plan in place without changing state or executing it; completed plans remain immutable. Guarded replacement may instead retire a source after successor creation. Material discoveries always require a new human decision. |

## Validation Source Notes

Keep source notes terse and machine-updatable:

- `validator required-related-link warnings` means
  `tools/skills_manager.py validate` checks `REQUIRED_RELATED_SKILL_LINK_RULES`:
  a small explicit rule list that can target skill names and structured taxonomy
  categories. Missing `SKILL.md` links are warnings for first-party skills only;
  ignored or lockfile-owned third-party skills are skipped.
- `taxonomy ... scenario` points to documented acceptance criteria in
  `skill-taxonomy.md`.
- `taxonomy ... boundary` points to the boundary tables in `skill-taxonomy.md`.
- `first-party skill frontmatter/body` should be used only when the route is not
  yet covered by taxonomy, repository docs, or validator rules. Third-party
  runtime skill bodies are not repository-owned routing authority.
